# 上流報告: @nuxtjs/tailwindcss v6 のHMR無限再生成ループ

- **報告先**: https://github.com/nuxt-modules/tailwindcss/issues (新規issue)
- **関連する既存issue**: [#877 "HMR triggers full page load on component edit"](https://github.com/nuxt-modules/tailwindcss/issues/877) — 同じ症状クラスの報告(2024年7月起票、open)。ただし根本原因は未特定のまま。新規issueを立てず#877にコメントとして貼る選択もあり
- **ステータス**: 下書き(未報告)
- **社内での対処**: patch-package で `app/patches/@nuxtjs+tailwindcss+6.14.0.patch` を適用済み(タイムスタンプ除去の2行修正)。上流で修正されたらパッチとバージョンピン留めを撤去する

## 日本語サマリ

`@nuxtjs/tailwindcss` 6.12.0以降、生成テンプレート(`.nuxt/tailwind/postcss.mjs`)の1行目に生成時刻(`new Date().toLocaleString()`)が埋め込まれる。一方、同モジュールは `app:templatesGenerated` フックで自テンプレートの変更を検知すると100ms後に `updateTemplates()` で再生成する。時刻は秒単位で変わるため「再生成 → 内容が変わった → また再生成」の無限ループになる。コンポーネントを1回編集するだけでループが始まり、`plugins.client.mjs` 起因のfull-reloadが連発してHMRが実質壊れ、最終的にdevサーバーが落ちる。タイムスタンプはどのコードからも参照されない純粋な情報表示用コメントであり、除去するとループが完全に消えることを実証済み。

## GitHub issue 下書き(以下をコピペ)

**Title:**

```
Infinite template regeneration loop: timestamp comment in generated config makes every regeneration a "change" (root cause of #877?)
```

**Body:**

```markdown
### Environment

- Operating System: Windows 11 (10.0.26200)
- Node Version: v24.14.1
- Nuxt Version: 3.15.0
- Nitro Version: 2.13.4
- Vite Version: 6.4.3
- Package Manager: npm
- Runtime Modules: @nuxtjs/tailwindcss@6.14.0

### Describe the bug

Editing **any watched file once** (e.g. adding one CSS line inside a component's
`<style>` block) starts a **self-sustaining infinite template regeneration loop**
that never stops:

- `.nuxt/tailwind/postcss.mjs` is rewritten roughly once per second, forever
- each cycle broadcasts `full-reload` (triggered by `virtual:nuxt:…/.nuxt/plugins.client.mjs`)
  to all HMR clients, so connected pages keep reloading and HMR is effectively broken
- Nuxt DevTools RPC `refresh("getComponents")` messages are broadcast at
  ~40–150 msg/s over the HMR WebSocket (we logged **88,000+ messages in ~80 minutes**
  from a single one-line edit), until the dev server eventually became unresponsive
  and died

Dev server log after a single one-line CSS edit:

    ℹ hmr update /components/…/ChatInputForm.vue?vue&type=style&index=0&scoped=…&lang.css
    ℹ page reload virtual:nuxt:…/.nuxt/plugins.client.mjs
    ℹ page reload virtual:nuxt:…/.nuxt/plugins.client.mjs (x2)
    ℹ page reload virtual:nuxt:…/.nuxt/plugins.client.mjs (x3)
    …continues forever…

### Root cause

The generated config template embeds a **generation timestamp** in its first line
([`src/internal-context/load.ts:222` at v6.14.0](https://github.com/nuxt-modules/tailwindcss/blob/v6.14.0/src/internal-context/load.ts#L222),
introduced in v6.12.0 by #818):

```js
`// generated by the @nuxtjs/tailwindcss <…> module at ${(new Date()).toLocaleString()}`,
```

Meanwhile the module regenerates its own template whenever it sees it in the
changed-templates list (dist/module.mjs of 6.14.0, same logic in source):

```js
// dist/module.mjs:455-467
const reloadConfigTemplate = async () => {
  …
  setTimeout(async () => {
    await updateTemplates({ filter: (t) => t.dst === dst || … });
    await nuxt.callHook("tailwindcss:internal:regenerateTemplates", …);
  }, 100);
};
nuxt.hook("app:templatesGenerated", async (_app, templates) => {
  if (…templates include the tailwind config template…) {
    await reloadConfigTemplate();
  }
});
```

The two combine into a feedback loop:

1. any watched-file change makes Nuxt regenerate templates
2. the tailwind template's content now differs (the timestamp has ticked to the
   next second), so Nuxt marks it as *changed* and fires `app:templatesGenerated`
3. the module's hook sees its template in the changed list → `updateTemplates()`
   again → new timestamp → back to step 2, forever

Each iteration also invalidates non-hot-acceptable virtual modules
(`plugins.client.mjs`), which is what turns every iteration into a client
`full-reload`.

I verified the diagnosis by snapshotting `.nuxt/tailwind/postcss.mjs` twice during
the loop — **the only difference between generations is the timestamp**:

```diff
-// generated by the @nuxtjs/tailwindcss <…> module at 2026/7/9 0:59:57
+// generated by the @nuxtjs/tailwindcss <…> module at 2026/7/9 0:59:59
```

The timestamp is purely informational: nothing in the package reads or parses it,
and it plays no role in cache invalidation (Vite invalidation is done explicitly
via `server.moduleGraph.invalidateModule`).

### Fix (verified)

Removing the date from the comment makes the template content deterministic and
completely eliminates the loop. With this one-line change (applied via
patch-package to both dist builds), the same edit produces exactly one clean
`hmr update`, zero `page reload`s, and no regeneration loop:

```diff
-          `// generated by the @nuxtjs/tailwindcss <https://github.com/nuxt-modules/tailwindcss> module at ${( new Date()).toLocaleString()}`,
+          `// generated by the @nuxtjs/tailwindcss <https://github.com/nuxt-modules/tailwindcss> module`,
```

Happy to open a PR for this.

### Reproduction

Observed in a real-world project (Nuxt 3.15.0, `ssr: false`, `srcDir: 'src/'`,
a `tailwind.config.js` at the project root, tailwindcss 3.4.x). Reproduced with
both `exposeConfig: true` and `false`. Trigger: start `nuxt dev`, load a page,
edit any component once.

Since the mechanism is deterministic (timestamped template content + the
`app:templatesGenerated` → `updateTemplates` hook above), it should reproduce in
any project where the config template ends up in the changed-templates list once.

### Additional context

This looks like the root cause of #877 (full page reloads on component edits,
reported against 6.12.2 nightlies — the first releases after #818 introduced the
timestamp). The `main` branch (v7 / Tailwind v4) no longer generates config
templates, so only the v6 line is affected.
