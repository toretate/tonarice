# チャットの処理の流れ

本ドキュメントでは、ユーザーからのチャット入力がどのように処理され、LLMのプロンプトに組み込まれ、ツールの実行（Tool-use）を経て最終的な応答（UI表示およびTTS音声）に加工されるかの詳細な流れを定義します。

---

# プロンプトの作成

ユーザーからチャットのメッセージを受け取った後、LLMへの入力プロンプトは以下の手順で作成されます。

1. **キャラクター設定プロンプトの読み込み**
    - マスコット固有のキャラクター設定（`mascot.config.yaml` 内の性格プロンプトや, 別途指定された `systemPrompt`）を取得します。

2. **動的コンテキスト（システム日時など）の付与**
    - 現在のシステム時刻を LLM が正しく認識できるように、以下の形式で日時指示をシステムプロンプトの末尾に動的に追加します。
      ```
      [現在のシステム日時]
      YYYY/MM/DD HH:mm:ss
      ```

3. **ツール使用ガイドラインの結合**
    - システムプロンプトの最下部に、使用可能なツール一覧（`getCurrentTime`, `searchTasks`, `addTask` など）と、ツール使用の判断基準、および最終回答に出力すべきでない特殊タグ（Thinking Process や `<|channel>` タグなど）の制限ルールを定義した「ツール使用ガイドライン」を結合します。
    - **プロンプトファイルの構造**:
      ガイドライン定義は、コードの変更を伴わずに調整できるよう、[app/src/skills/tool-use/prompts/](../../app/src/skills/tool-use/prompts/) ディレクトリ配下の独立した TypeScript モジュール（`*.prompt.ts`）に分割されています。エイリアス `@prompt/` を用いて、チャットAIサービスから読み込まれます。
    - **動的組み立て**:
      ユーザー設定でONになっている有効なツールに応じて、対応するツールの個別プロンプト（例: `getWeather.prompt.ts`, `manageTasks.prompt.ts` など）を動的に結合し、共通ガイドラインテンプレートである `tool-use-guideline.ts` の `{{toolUseSection}}` プレースホルダーを置換して完成させます。
    - 最終的なシステムプロンプト：
      `finalSystemPrompt = systemPrompt + 日時指示 + ツール使用ガイドライン`

4. **対話履歴（History）のマッピング**
    - 過去のチャットログを Vercel AI SDK の `CoreMessage[]` 形式（`role` や `content` を持つオブジェクト配列）にマッピングします。
    - **ダミーユーザーメッセージ of 挿入**:
      APIの制約（履歴の最初が `user` でなければならない等）による履歴の切り捨てやエラーを防止するため、履歴の最初のメッセージが `user` 以外（例: `assistant`）の場合は、先頭にダミーの `user` メッセージを自動で挿入します。

# LLMからの tool-use の処理の流れ

LLM が外部ツールを呼び出す必要があると判断した場合、以下の流れで処理が行われます。

1. **ツール呼び出し要求の検出**
    - Vercel AI SDK の `generateText` の推論時に、ツールコールが検出されます。

2. **サーバー側での実行インターセプト (`onToolExecute`)**
    - WebSocket サーバー（`ws.ts`）内の `onToolExecute` コールバックによってツールの実行が検知され、ローカルDB（`tasks.json`）の操作などが行われます。
        - **`searchTasks`**: 指定された条件でタスクを検索し、JSON形式で結果を返します。
        - **`updateTask`**: 指定されたタスクIDの状態を更新します。
        - **`deleteTask`**: 指定されたタスクIDのタスクを削除します。

3. **自動マルチステップによる実行と完了 (`stopWhen`)**
    - Vercel AI SDK v6 (6.x以降) の仕様に追従し、`generateText` の呼び出し時に `stopWhen: stepCountIs(5)` を設定しています。
    - これにより、LM Studio やローカルLLMを含むエンジンにおいて、ツール実行結果が SDK によって自動的に対話履歴 (`ModelMessage[]`) にマッピングされ、最大 5 ステップにわたり再帰推論が自動で継続されます。
    - 以前存在していた「手動での while ループ再帰処理」や「平文プロンプトインジェクション」によるハック的なワークアラウンドは、Zod スキーマ検証（引数の `args` キー、および `tool-result` パートにおける `output: { type: 'json', value }` の厳密なマッピング）に完全対応したことで、すべて撤去されました。

4. **クライアントとの同期およびDB反映 (`onToolResult`)**
    - ツールの実行完了が通知されると `onToolResult` がトリガーされます。
    - 新しいタスクの追加（`addTask` / `addSchedule`）などの場合、サーバー側で `addTaskToDb` を実行し、クライアントに `task-action` イベントを送信してUI側とデータを同期します。

# LLMからのチャット返答の加工処理の流れ

LLM から最終的なテキスト応答が得られた後、UIや音声（TTS）に渡される前にテキストがパースおよび加工されます。

1. **タイマー制御タグの処理**
    - LLM の応答に `[TIMER:秒数,メモ]` というタグが含まれている場合、正規表現でこれを抽出して `setTimeout` でタイマーを登録します。登録後、応答テキストからタグを除去します。

2. **感情タグの処理**
    - LLM の応答から感情を示すタグ（例: `[喜び]`, `[悲しみ]`）を正規表現 `/\[(\w+)\]/` で抽出（`detectedEmotion`）し、表情切り替え用の情報とします。
    - 抽出後、応答テキストから感情タグを除去してクリーンな `speechText` にします。

3. **UI表示用のテキストの加工**
    - 感情タグやタイマー指示タグが削除されたテキストが、`chat-response` イベントを介してチャットウィンドウのバブルに表示されます。
    - AI応答はMarkdownとして解釈し、見出し、段落、強調、斜体、打ち消し、箇条書き、番号付きリスト、引用、水平線、コード、表、リンクを構造化して表示します。
    - Markdownは生HTMLへ変換せずVue要素として描画します。HTMLタグは文字列として表示し、リンクは `http` / `https` / `mailto` のみクリック可能にします。
    - 狭いチャット幅では文章を折り返し、表とコードブロックだけを水平スクロール可能にします。

4. **TTS 送信前の正規化（TTS Normalizer）**
    - 音声合成APIに送られる直前で、読み上げに適さない記号や英単語などを日本語読みに正規化する `normalizeTextForTts` 処理が実行されます（チャット応答、タスク通知、タイマー通知、直接TTS APIの全4経路に適用）。
    - **正規化処理の順序と仕様**:
        1. **マークダウンマーカー除去**: `**bold**` → `bold`、`__bold__` → `bold`、`` `code` `` → `code`、`~~strike~~` → `strike`、`[表示](url)` → `表示` のように、装飾記号やリンクURL部分を除去し、テキスト内容のみを残します。ただし、単一のアスタリスク（`*`）とアンダースコア（`_`）は、行動描写の判定や識別子の保護のために除去しません。感情タグ（`[happy]`など）も不変のまま保持します。
        2. **日付の日本語読み変換**: スラッシュ区切りの日付（`2026/07/13` → `2026年7月13日`、`7/13` → `7月13日`）、ISOハイフン日付（`2026-07-13` → `2026年7月13日`）、全角スラッシュ（`7／13` → `7月13日`）を日本語表記に変換します。月日における先頭のゼロは除去されます（`07/03` → `7月3日`）。半角数字のみが対象です。※月日のみのハイフン日付（`7-13`）やCIDR・IP、全角数字は対象外です。なお分数表記（例 `1/2`）は日付（`1月2日`）として読み上げられます（既知の制限。タスク・予定の文面では日付が大多数のため受容）。
        3. **時刻の日本語読み変換**: `11:00` → `11時`（分が `00` のときは分を省略）、`11:15` → `11時15分` のように変換します。全角コロン（`11：00` → `11時`）も対象です。半角数字のみが対象です。※IPのポート番号や `11:22:33` などの3連表記、全角数字は対象外です。
        4. **英単語カタカナ置換（ハイブリッド辞書）**:
            - アプリ共通のIT・略語辞書（`tts-it-overrides.json`）やカスタム辞書（`tts-custom-overrides.json`）に加えて、マスコットデータ（`MascotData.aiConfig.ttsDictionary`、electron に永続化される）で指定された個別辞書を後勝ち（個別が最優先）でマージします。
            - 置換は1本の結合正規表現を用いた単一パス（1パス）で実行されるため、連鎖置換は発生しません。
            - トークンの末尾記号（`.`, `'`, `-` など）を考慮するため、文末の `API.` などのマッチングも正しく置換されます。
            - 辞書の**値**に英数字やマークダウン記号、日付/時刻区切り記号を含むエントリは、二重正規化時（クライアントでの一時適用→サーバーAPI呼び出しなど）の冪等性を保証するために自動的に**無視**されます。これにより、何度処理を通しても結果が崩れない（冪等性）ことが構造的に担保されます。

5. **TTS に渡すときのテキストの加工（台詞の抽出）**
    - `ttsReadNarrative`（デフォルト `false`）が `false` の場合、`filterDialogue` 関数を通して以下の詳細ルールに基づき加工を行い、発話（セリフ）のみを抽出します。ロールプレイ的な応答（地の文＋「」のセリフ）か、通常の相談的な応答（括弧なしの地の文のみ）かは、このルール自体が応答テキストの括弧パターンから都度自動判定するため、ユーザーやプロンプト側でモードを明示的に切り替える必要はありません。
    - `ttsReadNarrative` を `true`（朗読モード）にした場合のみ、このフィルタをスキップして地の文を含む全文をそのまま読み上げます。
        - **行動描写の除去**: 丸括弧 `（...）`, `(...)` や アスタリスク `*...*` で囲まれた行動描写は、発話しないト書きとしてテキストから完全に除外します。
        - **引用台詞の判定（地の文の切り落とし）**:
          `「...」` や `『...』` などの括弧が存在し、かつ**括弧の直後が台詞の引用を伴う特定の助詞・動詞**（`と`、`って`、`と言`、`と呟`、`と囁`、`と叫`、`と微笑` など）で始まっている場合のみ、括弧の外側を「読み上げ不要な地の文」と判定し、括弧内のみを抽出して結合します。
          * 例: `「了解！」と嬉しそうに頷く。` → `了解！`
        - **強調・連続セリフの維持**:
          括弧の直後が上記の引用パターンに該当しない場合、括弧は単なる言葉の強調または通常のセリフとして扱われます。この場合、**括弧記号（「」『』）のみを除去**し、括弧の外側（「〜」を含む文章など）も含めた全体のテキストをすべて読み上げ対象として維持します。
          * 例: `はーい、了解です〜！「〇〇を登録したよ！」` → `はーい、了解です〜！〇〇を登録したよ！`（全体を読み上げる）
    - **適用経路の処理順**:
      チャット応答時、音声用テキストは以下の順序で加工されたのち、文単位に分割（`splitSentences`）されてVOICEVOX / Irodori TTSへ送られます。
      `normalizeTextForTts` → `filterDialogue` → `stripResidualAsterisks` → `splitSentences`
      （※アスタリスク文字自体の除去 `stripResidualAsterisks` は、`filterDialogue` の判定完了後の最後に適用されます）
    - 合成された音声バイナリは Base64 形式で `chat-audio` イベントを介してクライアントに送信され、UI側で再生されます。

# システムプロンプト構築の今後の改善

## プロンプトキャッシュ（Prompt Caching）の最適化

現状の実装では、システムプロンプト（静的情報）の中央に「秒単位で変化する動的なシステム日時（`timeInstruction`）」が直接インジェクトされています。
```
finalSystemPrompt = systemPrompt (静的) + 日時指示 (動的) + ツール使用ガイドライン (静的)
```

この構築方式には、LLM API 側の**プロンプトキャッシュ（Prompt Caching）**の観点から以下の課題があります。

1. **キャッシュミスの発生**:
   プレフィックス（入力テキストの先頭部分）が一致していればAPIコストやレスポンスの生成開始時間を大幅に削減できますが、日時が秒刻みで変動するため、チャット要求のたびにキャッシュ全体が破棄（キャッシュミス）されてしまいます。
2. **静的情報のキャッシュ無効化**:
   `日時指示` の後ろにある `ツール使用ガイドライン` のような静的で長いガイドライン定義テキストも、日時の変更によって毎ターン再処理されるため、キャッシュの恩恵を一切受けられません。

### 推奨される改善案

- **静的なシステムプロンプトと動的情報の分離**:
  `systemPrompt` と `ツール使用ガイドライン` は「静的なシステムプロンプト（`system` オプション）」として一貫してキャッシュ可能にします。
- **日時情報の配置変更**:
  動的に毎ターン変動する `日時指示` は、システムプロンプトのコンテキスト（`system`）から除外し、対話履歴 `messages` の**最後の `user` メッセージの直前**（または最新のターンのシステムプレフィックス）にシステム情報として割り込ませます。
- **キャッシュ制御タグの明示指定**:
  API（Gemini などの Context Caching や Anthropic の Prompt Caching）が対応している場合、静的なプロンプト境界にキャッシュ指定用フラグ（`cache_control: { type: "ephemeral" }` など）を明示的に付与して呼び出す設定をプロバイダーアダプター側に追加します。

これらにより、チャット応答の生成速度（ファーストトークン遅延）が劇的に改善され、API の利用料金コストも抑えることが可能になります。
