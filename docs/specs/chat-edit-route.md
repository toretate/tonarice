# チャットの処理の流れ

本ドキュメントでは、ユーザーからのチャット入力がどのように処理され、LLMのプロンプトに組み込まれ、ツールの実行（Tool-use）を経て最終的な応答（UI表示およびTTS音声）に加工されるかの詳細な流れを定義します。

---

# プロンプトの作成

ユーザーからチャットのメッセージを受け取った後、LLMへの入力プロンプトは以下の手順で作成されます。

1. **キャラクター設定プロンプトの読み込み**
    - マスコット固有のキャラクター設定（`mascot.config.yaml` 内の性格プロンプトや, 別途指定された `systemPrompt`）を取得します。

2. **動的コンテキスト（システム日時など）の付与**
    - 現在のシステム時刻を LLM が正しく認識できるように、以下の形式で日時指示をシステムプロンプトの末尾に動的に追加します。
      ```
      [現在のシステム日時]
      YYYY/MM/DD HH:mm:ss
      ```

3. **ツール使用ガイドラインの結合**
    - システムプロンプトの最下部に、使用可能なツール一覧（`getCurrentTime`, `searchTasks`, `addTask` など）と、ツール使用の判断基準、および最終回答に出力すべきでない特殊タグ（Thinking Process や `<|channel>` タグなど）の制限ルールを定義した「ツール使用ガイドライン」を結合します。
    - 最終的なシステムプロンプト：
      `finalSystemPrompt = systemPrompt + 日時指示 + ツール使用ガイドライン`

4. **対話履歴（History）のマッピング**
    - 過去のチャットログを Vercel AI SDK の `CoreMessage[]` 形式（`role` や `content` を持つオブジェクト配列）にマッピングします。
    - **ダミーユーザーメッセージ of 挿入**:
      APIの制約（履歴の最初が `user` でなければならない等）による履歴の切り捨てやエラーを防止するため、履歴の最初のメッセージが `user` 以外（例: `assistant`）の場合は、先頭にダミーの `user` メッセージを自動で挿入します。

---

# LLMからの tool-use の処理の流れ

LLM が外部ツールを呼び出す必要があると判断した場合、以下の流れで処理が行われます。

1. **ツール呼び出し要求の検出**
    - Vercel AI SDK の `generateText` の推論時に、ツールコールが検出されます。

2. **サーバー側での実行インターセプト (`onToolExecute`)**
    - WebSocket サーバー（`ws.ts`）内の `onToolExecute` コールバックによってツールの実行が検知され、ローカルDB（`tasks.json`）の操作などが行われます。
        - **`searchTasks`**: 指定された条件でタスクを検索し、JSON形式で結果を返します。
        - **`updateTask`**: 指定されたタスクIDの状態を更新します。
        - **`deleteTask`**: 指定されたタスクIDのタスクを削除します。

3. **自動マルチステップによる実行と完了 (`stopWhen`)**
    - Vercel AI SDK v6 (6.x以降) の仕様に追従し、`generateText` の呼び出し時に `stopWhen: stepCountIs(5)` を設定しています。
    - これにより、LM Studio やローカルLLMを含むエンジンにおいて、ツール実行結果が SDK によって自動的に対話履歴 (`ModelMessage[]`) にマッピングされ、最大 5 ステップにわたり再帰推論が自動で継続されます。
    - 以前存在していた「手動での while ループ再帰処理」や「平文プロンプトインジェクション」によるハック的なワークアラウンドは、Zod スキーマ検証（引数の `args` キー、および `tool-result` パートにおける `output: { type: 'json', value }` の厳密なマッピング）に完全対応したことで、すべて撤去されました。

4. **クライアントとの同期およびDB反映 (`onToolResult`)**
    - ツールの実行完了が通知されると `onToolResult` がトリガーされます。
    - 新しいタスクの追加（`addTask` / `addSchedule`）などの場合、サーバー側で `addTaskToDb` を実行し、クライアントに `task-action` イベントを送信してUI側とデータを同期します。

# LLMからのチャット返答の加工処理の流れ

LLM から最終的なテキスト応答が得られた後、UIや音声（TTS）に渡される前にテキストがパースおよび加工されます。

1. **タイマー制御タグの処理**
    - LLM の応答に `[TIMER:秒数,メモ]` というタグが含まれている場合、正規表現でこれを抽出して `setTimeout` でタイマーを登録します。登録後、応答テキストからタグを除去します。

2. **感情タグの処理**
    - LLM の応答から感情を示すタグ（例: `[喜び]`, `[悲しみ]`）を正規表現 `/\[(\w+)\]/` で抽出（`detectedEmotion`）し、表情切り替え用の情報とします。
    - 抽出後、応答テキストから感情タグを除去してクリーンな `speechText` にします。

3. **UI表示用のテキストの加工**
    - 感情タグやタイマー指示タグが削除されたテキストが、そのまま `chat-response` イベントを介してチャットウィンドウのバブルに表示されます。

4. **TTS に渡すときのテキストの加工（台詞の抽出）**
    - キャラクター設定で「地の文を読み上げない（`ttsReadNarrative === false`）」に設定されている場合、`filterDialogue` 関数を通して以下の加工を行います。
        - **行動描写の除去**: 丸括弧 `（...）`, `(...)` や アスタリスク `*...*` で囲まれた行動描写をテキストから除外します。
        - **台詞の抽出**: `「...」` や `『...』` などの括弧で囲まれた部分がある場合、括弧の外側に言葉（地の文の引用など）があるか判定し、中身のセリフ部分だけを抽出して結合します。単語の強調のための括弧などの場合は、括弧のみを外して全体を読み上げます。
    - 加工された音声用テキストは、文単位に分割（`splitSentences`）され、VOICEVOX または Irodori TTS などの音声エンジンへ並行で音声合成リクエストが送られます。
    - 合成された音声バイナリは Base64 形式で `chat-audio` イベントを介してクライアントに送信され、UI側で再生されます。

# Zod バリデーションエラーの解決と自動継続の実現（設計記録）

開発の初期フェーズにおいて、Vercel AI SDK の Zod スキーマ検証エラーを回避するため「平文プロンプトインジェクション」方式を一時的に採用していましたが、検証ブランチにおける調査によって不整合の原因がすべて解明され、AI SDK v6 の正式仕様に完全に準拠した**「正規の解決（標準対話履歴モデルへの一本化）」**が完了しました。

## 解決した Zod エラーの要因と対策

1. **1回目リクエストの Bad Request 解消（`inputSchema` への移行と Zod 委譲）**
    - **課題**: ツール定義のパラメータキーとして `@lmstudio/sdk` に合わせた `parameters` を使用していたため、AI SDK の `generateText` 内部で `inputSchema` が空の `{"properties":{}}`（`type` 欠落）になり、LM Studio 側で 400 エラーが発生していました。また、自前の `convertLmStudioSchemaToPlainJsonSchema` 関数による JSON Schema 変換では `z.enum` や `z.array`、オブジェクトのネストなどがすべて `string` 型に潰されてしまう問題もありました。
    - **解決**: ツール定義のキー名を AI SDK 正式の **`inputSchema`** に統一し、さらに自前の JSON Schema 変換処理を完全に廃止しました。LM Studio SDK から提供される ZodObject（`lmTool.parametersSchema`）を **`inputSchema` に直接引き渡す**設計に修正したことで、AI SDK 内部の `zodSchema()` 処理にスキーマ生成を委譲でき、`z.enum` や `z.array`、ネストしたオブジェクトが正しい表現のまま LM Studio へ送信されるようになりました。

2. **2回目手動再帰時の `args` 不整合解消**
    - **課題**: LLM のレスポンス（`toolCalls`）から返る引数キーは `input` であったため、手動履歴構築でそのまま渡すと Zod が期待する `args` が `undefined` になり ZodError が発生していました。
    - **解決**: 履歴構築時に `args: tc.args || tc.input || {}` のマッピングを施すことで解決しました（※手動 loops 廃止に伴い現在は自動解決されています）。

3. **`tool-result` メッセージの `output` 構造不整合解消**
    - **課題**: AI SDK v6 (6.x以降) での `LanguageModelV1Prompt` 内部スキーマの厳密化に伴い、`role: 'tool'` メッセージの各パートには `output` プロパティが必須であり、かつ `{ type: 'json', value: ... }` のようなディスクリミネーターユニオンの構造を満たす必要がありました。
    - **解決**: `output: { type: 'json', value: parsedVal }` の形で明示的マッピングを施すことで解決しました（※手動 loops 廃止に伴い現在は自動解決されています）。

4. **自動継続機能の正常化（`maxSteps` から `stopWhen` への置換）**
    - **課題**: v5/v6 で `maxSteps` オプションが廃止されたため、自動継続が機能せず `finishReason: 'tool-calls'` で処理が止まり、結果として手動 loops や `steps` の手動マージ（ゲッタープロパティの破壊危険処理）が発生していました。
    - **解決**: `stopWhen: stepCountIs(5)` に置き換えることで、SDK 側で安全に自動継続マルチステップを完結させられるようになりました。

これらにより、以前の「平文プロンプトインジェクション」および「手動 while ループ」のワークアラウンドはすべて安全に撤去され、本来の対話履歴である `ModelMessage[]` の健全性を維持したクリーンな実装に一本化されました。

> [!TIP]
> **コネクター（接続レイヤー）の役割と今後の方向性**:
> LM Studio などのローカル API サーバーとの通信仕様の違いやツールコールの解釈における非互換性は、Vercel AI SDK の **Provider Adapter（接続コネクター）レイヤー**（`createOpenAI` などでのエンドポイント指定やカスタムコネクターの実装）で吸収されるべき問題です。
> 
> 今回の修正によって、アプリケーションレイヤーのコードから一時的なハックやインジェクションが完全に排除され、プロトコル変換をプロバイダーおよびスキーマ適合処理（`tool-adapter.ts`）に局所化できたことは、設計の健全性と保守性の観点から非常に大きな意義を持ちます。特に、不要な自前スキーマ変換の廃止と AI SDK への直接委譲により、将来的に複雑なスキーマ（列挙値や配列）が追加されても自動で正しく認識・送信される土台が整いました。

# システムプロンプト構築の今後の改善

## プロンプトキャッシュ（Prompt Caching）の最適化

現状の実装では、システムプロンプト（静的情報）の中央に「秒単位で変化する動的なシステム日時（`timeInstruction`）」が直接インジェクトされています。
```
finalSystemPrompt = systemPrompt (静的) + 日時指示 (動的) + ツール使用ガイドライン (静的)
```

この構築方式には、LLM API 側の**プロンプトキャッシュ（Prompt Caching）**の観点から以下の課題があります。

1. **キャッシュミスの発生**:
   プレフィックス（入力テキストの先頭部分）が一致していればAPIコストやレスポンスの生成開始時間を大幅に削減できますが、日時が秒刻みで変動するため、チャット要求のたびにキャッシュ全体が破棄（キャッシュミス）されてしまいます。
2. **静的情報のキャッシュ無効化**:
   `日時指示` の後ろにある `ツール使用ガイドライン` のような静的で長いガイドライン定義テキストも、日時の変更によって毎ターン再処理されるため、キャッシュの恩恵を一切受けられません。

### 推奨される改善案

- **静的なシステムプロンプトと動的情報の分離**:
  `systemPrompt` と `ツール使用ガイドライン` は「静的なシステムプロンプト（`system` オプション）」として一貫してキャッシュ可能にします。
- **日時情報の配置変更**:
  動的に毎ターン変動する `日時指示` は、システムプロンプトのコンテキスト（`system`）から除外し、対話履歴 `messages` の**最後の `user` メッセージの直前**（または最新のターンのシステムプレフィックス）にシステム情報として割り込ませます。
- **キャッシュ制御タグの明示指定**:
  API（Gemini などの Context Caching や Anthropic の Prompt Caching）が対応している場合、静的なプロンプト境界にキャッシュ指定用フラグ（`cache_control: { type: "ephemeral" }` など）を明示的に付与して呼び出す設定をプロバイダーアダプター側に追加します。

これらにより、チャット応答の生成速度（ファーストトークン遅延）が劇的に改善され、API の利用料金コストも抑えることが可能になります。

## ツール数（プロンプトサイズ）の最適化

LLM のツール呼び出し（Tool-use）機能において、提供するツールの数やスキーマ定義が肥大化すると、プロンプトのトークン消費量が増え、モデルが誤ったツールを呼び出す確率（Hallucination）も高くなります。以下の2点によるプロンプトの軽量化と呼び出し精度の向上が推奨されます。

### 1. 【廃止済み】`getCurrentTime` ツールの廃止
* **対応内容**: システム時刻を取得する `getCurrentTime` ツールを完全に廃止し、LLMに公開されるツール群のスキーマ定義、設定画面のUI（チェックボックス）、および関連するテストコードから削除しました。
* **理由**: プロンプト作成時に「現在のシステム日時」がテキスト情報として常に LLM へ動的にインジェクトされているため、LLM が現在日時を知る目的でわざわざ外部ツールを呼び出す必要はありません。この無駄なツール定義を排除することで、プロンプトのトークンサイズ削減と誤呼び出しの防止（Hallucinationの低減）を図りました。


### 2. 【実装済み】Task / Schedule 操作ツールの一元化 (`manageTasks`)
* **構成**: `addTask`, `searchTasks`, `updateTask`, `deleteTask`, `addSchedule` の5つの操作を **`manageTasks`** という単一の汎用タスク管理ツールに一元化しました。
* **メリット**:
  - LLMが読み込む JSON スキーマ全体の記述量を約 1/3 以下に圧縮し、対話時のコンテキストウィンドウ（トークン消費量）を大幅に節約しました。
  - 個別ツールの使い分けによる誤判定を減らし、指示に対するツール呼び出し（Tool-use）の成功精度を高めました。
* **スキーマ設計 (Zod)**:
  ```typescript
  export const manageTasksToolShared = tool({
      name: 'manageTasks',
      description: 'タスク（TODO）やスケジュール（予定）の追加・検索・更新・削除を行います。期限のないタスクは add で scheduledAt を省略し、日時指定のある予定は add で scheduledAt を指定してください。',
      parameters: {
          action: z.enum(['add', 'search', 'update', 'delete']).describe('実行する操作。add=追加、search=検索、update=更新、delete=削除。'),
          title: z.string().optional().describe('タスク・予定のタイトル（add 時は必須）'),
          scheduledAt: z.string().nullable().optional().describe('予定日時。ISO 8601形式の文字列 (例: 2026-07-06T18:00:00+09:00)。add 時に指定すると期限付きの予定になり、省略すると期限なしタスクになる。update 時に空文字列 "" または null を指定すると期限を削除できる。'),
          priority: z.enum(['normal', 'star', 'thunder']).optional().describe('優先度。デフォルトは normal。'),
          categoryId: z.string().optional().describe('追加先のカテゴリID。Work（仕事）の場合は default、Private（プライベート）の場合は private を指定してください。'),
          id: z.string().optional().describe('操作対象のタスクID（update・delete 時は必須）'),
          query: z.string().optional().describe('タイトルに含まれる検索キーワード（search 時）'),
          date: z.string().optional().describe('検索対象 of 予定日。YYYY-MM-DD形式の文字列（search 時）'),
          completed: z.boolean().optional().describe('search 時は完了状態で絞り込む条件（true=完了, false=未完了, 未指定=両方）。update 時は更新後の完了状態。')
      },
      ...
  ```
