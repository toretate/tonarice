# マスコット表情スプライト自動位置合わせ仕様書

## 1. 概要

### 1.1 背景

マスコット画像から、顔部分を拡大して最大16種類の表情スプライトを作る処理は作成済みです。

* マスコット詳細 - 表情アセット - AI表情作成 を実行

このスプライト作成後、16種類の表情スプライトを区分けして、各表情スロットに設定する処理も作成済みです。

* マスコット詳細 - 表情アセット - AIスプライトインポート を実行

作成されたスプライトをそのまま使うには下記の問題があります。

* 表情ラベルや、区分けに使った枠が残る
* 眉、目、口の表情が必要だが、不要な部分まで含まれている
* 表情画像の拡大率がマスコット画像と違うため、拡大もしくは縮小しないと合わない

この自動位置合わせは、下記の手順を実行することで自動的に表情スプライトを最適な位置・拡大率に合わせます。

1. 表情スプライトとマスコットの顔位置の把握
2. 表情スプライトの拡大 / 縮小（scale）と回転（rotation）の算出
3. 表情スプライトの位置（offset）設定
4. 顔以外の不要領域のマスク

### 1.2 設計の基本方針（重要）

本仕様は、初版の素案（「表情スプライト1枚ごとに、輝度ベースの目検出で個別に位置合わせする」）から、以下の理由で設計を見直したものである。

* **位置合わせは「表情1枚ごと」ではなく、原理的に「シート（衣装×ポーズ）単位で共通」である。** 16表情はすべて同一のマスコット顔から生成された同一構図・同一スケールであるため、本来 (scale, rotation, offset, mask) は1組で共有できる。
* **表情ごとの輝度ベース目検出は不安定。** 閉じ目・ウインク・色付き瞳・肌の濃淡で破綻し、表情ごとに scale がブレて切替時にガタつく。
* **対象はアニメ/イラストの顔であり、実写顔向けライブラリ（face-api.js 等）は使えない。**
* **AI API のランニングコストを避け、完全ローカルで処理する。**
* **このタスクは「別人の顔検出」ではなく「同一キャラの2枚の位置合わせ（レジストレーション）」であり、ランドマーク検出を回避できる。**

これらを踏まえ、以下の3つを設計の柱とする。

1. **責務を「共有変換の確立(A)」と「1枚の配置(B)」に分離する**（→ 第3章）
2. **OpenCV.js による画像レジストレーションで scale/rotation/offset を直接推定する（モデル不要・ローカル・決定論的）**（→ 第4章）
3. **正解データ `expr_*_OK.png` を目的関数とした自動評価ループを持つ**（→ 第9章）

## 2. 用語・前提

| 用語 | 定義 |
|---|---|
| ベース画像 | マスコットの全身/バストアップ画像（衣装・ポーズごと） |
| 顔領域 | ベース画像中の顔（眉・目・口・必要装飾を含む矩形/楕円） |
| 表情スプライト | AIが生成し、インポートで切り出された個別の表情画像 |
| シート単位 | 衣装×ポーズの1組。この単位で共有変換を持つ |
| 共有変換 (A) | scale / rotation / 顔マスク。シート単位で1組、キャッシュ対象 |
| 配置 (B) | offsetX / offsetY（＋必要なら表情別の微小Δ）。表情ごと |
| 相似変換 | 平行移動 + 回転 + 等方スケール（4自由度）。アフィンの部分集合 |

データ構造は既存の `MascotAsset`（`ui/src/mascots/mascot-asset.ts`）の `offsetX / offsetY / scale` を踏襲し、**`rotation?: number` を新規追加**する（第6章）。

## 3. 全体アーキテクチャ

### 3.1 責務の分離（A / B）

| 責務 | 中身 | 単位 | キャッシュ |
|---|---|---|---|
| **A. 共有変換の確立** | scale / rotation / 顔マスク | 衣装×ポーズで1組 | する |
| **B. 1枚の配置** | offsetX/Y（＋任意の微小Δ） | 表情ごと | しない |

* **マスコット側の顔情報**（顔領域・特徴点）は、どちらの実行モードでも必要で、**outfit/poseごとに1回算出してキャッシュ**する。これが両モード共通の土台となる。
* コアは純粋関数として実装する。

```text
solveTransform(マスコット顔情報, スプライト) → { scale, rotation, offsetX, offsetY, confidence }
```

* 同入力→同出力（決定論的）なので、キャッシュ・回帰テストが効く。

### 3.2 2つの実行モード

本処理は「16表情まとめて実行」と「1つだけ選んで実行」の両方を持つ。両者は同じ部品の組み合わせで表現する。

* **16枚まとめて実行**
  1. 共有変換 A を1回確立（基準は「通常(neutral)」スプライト ← 後述の既定）
  2. 全16枚に対して配置 B を適用

* **1枚だけ選んで実行**
  1. 共有変換 A が未確立ならその場で確立
  2. 選択された1枚にのみ配置 B を適用

すなわち「**まとめて = solveTransform を16回ループ、単体 = 1回呼ぶだけ**」であり、モード固有の特別扱いは持たない。

### 3.3 2モードの整合性を保証する設計

* 共有変換の **基準は「通常(neutral)」スプライト固定**（既定）。これにより単体実行とまとめ実行の結果が常に整合する。
* 単体実行で**基準以外の表情**を処理した場合、**共有 scale/rotation を上書きしない**（その表情の offset と任意の微小Δのみ書き込む）。
* 共有 scale/rotation の更新は、「通常」スプライトを処理したとき、または高確信度の基準からのみ行う。

### 3.4 パイプライン全体像

```text
[1回/シート・キャッシュ]
  ベース画像 ──▶ 顔領域 + 顔クロップ ─┐
                                      ├─▶ OpenCV.js レジストレーション ─▶ 共有変換 A
  通常スプライト(ラベル除去済) ───────┘     (scale, rotation, offset, confidence)
                                                       │
        ┌──────────────────────────────────────────────┘
        ▼
  全16表情 / 選択1枚 ── 共有Aを適用 + 配置B(offset, 任意Δ) ── 楕円フェザーマスク
        │
        ├─ confidence 低 → 手動エディタ(ExpressionEditorModal)へプリセット値を渡し確認
        │
        └─ expr_*_OK.png との IoU で自動評価（CI / 回帰）
```

## 4. 位置合わせアルゴリズム

### 4.1 技術選定の評価

| 選択肢 | 実行場所 | アニメ顔への適性 | 判定 |
|---|---|---|---|
| face-api.js | ブラウザ | ❌ 実写顔(WIDER/CelebA系)学習。アニメで不安定 | 非推奨 |
| transformers.js | ブラウザ | △ ONNX化されたアニメ用ランドマークモデルがほぼ無い | 賭けに弱い |
| **OpenCV.js** | **ブラウザ(wasm)** | ◎ レジストレーションに最適・モデル不要・決定論的 | **本命** |
| Python: lbpcascade_animeface | サーバ | ○ アニメ顔bbox検出は軽量だがランドマーク無し | 補助 |
| Python: anime-face-detector | サーバ | ◎ 28点ランドマーク。torch/mmpose依存が重く配布が重い | 最終手段(オプション) |

### 4.2 本命：OpenCV.js による特徴点レジストレーション（renderer・ローカル）

表情スプライトはマスコット自身の顔から生成された**同一キャラ**である。したがって「別人の顔を検出する」のではなく「**同じ顔の2枚を位置合わせする**」問題として解く。これによりアニメ用ランドマークモデルが無い問題を丸ごと回避する。

実行場所は **renderer（ブラウザ）側**とする。理由：Canvas/画像処理資産（`content-bounds-detector` `background-masker` 等）が renderer に揃っており、Python 同梱が不要で、OpenCV.js は wasm 8〜10MB を初回ロードでキャッシュするだけで済むため。

手順：

1. ベース画像から顔領域をクロップする（顔領域は 4.4 の方法で把握）。
2. 通常スプライトをラベル・区切り線除去済みの状態で用意する（インポート時の `box_2d` クロップ済み前提）。
3. `ORB` で両画像の特徴点を抽出 → `BFMatcher` + RANSAC でマッチング。
4. `cv.estimateAffinePartial2D()` で**相似変換（scale + rotation + translation）を直接取得**する。これが欲しい変換そのものになる。
5. 補助として `cv.findTransformECC()`（密なECC）でフォールバック / 微調整を行う。

* **ORB を主、ECC を従** とする。ORB は特徴ベースで画風差（AI生成による元画像との乖離）に比較的強い。ECC は密で高精度だが見た目依存が強いため微調整・フォールバック用。
* マッチ点数・RANSAC インライア率から **confidence** を算出する。

### 4.3 補助：改良版・特徴島検出によるクロスチェック（既存資産流用）

既存 `ui/src/skills/expression-alignment/feature-island-detector.ts` を流用しつつ改良する。

* **目の「中心座標」のみ**を用いる（中心間距離は表情で形が変わっても比較的不変）。輝度しきい値依存・閉じ目破綻といった現行の脆さを、scale 推定の主軸から外す。
* レジストレーション結果(4.2)とクロスチェックし、**大きく食い違えば confidence を下げる**。
* 単独では使わず、あくまで検証・フォールバック用とする。

> 注：現行コードはコメントと実値の不整合（輝度「160」とあるが実装は `110`、面積上限「1000」だが `1200`）があり、手調整による過適合の証跡。改良時に解消する。

### 4.4 顔領域の把握（ベース画像側）

* 既存 `face-region-detector.ts` のヒューリスティック（非透明領域の上部 9〜35%、幅中央50%）を初期推定として使用。
* レジストレーション(4.2)が成立すれば、顔領域は通常スプライト↔ベース画像の対応から逆算でき、ヒューリスティックの誤差を吸収できる。
* **Gemini Vision 等のクラウド顔検出はローカルonly方針のため既定では使用しない**（既存コードに残す場合も既定オフ）。

### 4.5 最終手段（オプション）：Python anime-face-detector

OpenCV.js が苦手な極端な構図（横向き・大きな傾き等、第8章）専用のオプション経路として、サーバ側に `anime-face-detector`（28点ランドマーク）を置く案を残す。

* **既定では導入しない。** torch/mmpose（数百MB〜GB級）の同梱はデスクトップ配布を著しく重くするため、レジストレーションで精度不足が確認された場合にのみ、同梱可否を判断して採用する。

## 5. 変換の解法（数式）

両目の中心（または ORB 対応点群）から相似変換を解く。

* スケール：`scale = |両目ベクトル_base| / |両目ベクトル_sprite|`
* 回転：`rotation = 2ベクトルの成す角`（→ 顔の傾きに自動対応）
* 平行移動：両目中点（または対応点重心）を一致させる
* 口を含めた3点以上があれば、`estimateAffinePartial2D` / 最小二乗で頑健化する。

最終的な値は、純粋なピクセル空間の変換として解いたうえで、**最後にエディタの offset/scale/rotation 値へ変換**する。プレビュー固定サイズ（420×560）や 140px 基準・マジックナンバー（現行の `+28/-32/+4` 等）への密結合を排し、解像度非依存・テスト可能にする。

## 6. データ構造の拡張（rotation）

`MascotAsset`（`ui/src/mascots/mascot-asset.ts`）に回転を追加する。

```typescript
interface MascotAsset {
    // 既存
    offsetX?: number;   // [-250, 250]
    offsetY?: number;   // [-250, 250]
    scale?: number;     // [0.3, 2.0]
    // 追加
    rotation?: number;  // 度。未指定は 0（後方互換）
}
```

* エディタ（`ExpressionEditorModal.vue`）とレンダラ（マスコット描画）が `rotation` を反映する。
* 既存データは `rotation` 未指定＝0 で後方互換を保つ。**フィールドだけ先行追加**し、横向き対応（第8章）は段階導入する。

## 7. マスク

* 初版素案の「矩形外をマスク」は、顔が丸いため角に首・髪・背景が残りやすい。
* **検出した両目・口に合わせた楕円マスク + 境界フェザー（ぼかし）**を採用する。
* 既存 `background-masker.ts` の soft / edge-smooth モードを流用可能。
* ラベル・区切り線はインポート時の `box_2d` クロップで除去済みである前提を明記する。
* マスクは共有変換 A の一部としてシート単位で持つ。

---

## 8. 確信度と手動フォールバック

* `solveTransform` は **confidence** を返す（ORB インライア率・ECC 相関・クロスチェック一致度から算出）。
* confidence が低い場合は**全自動で確定せず**、既存の手動エディタ `ExpressionEditorModal.vue` に**プリセット値（推定結果）を渡してユーザー確認**とする（「全自動」ではなく「自動下書き＋確認」UX）。
* 単体実行で閉じ目・ウインク・片目のみ等の不安定な表情を処理した場合：
  * キャッシュ済みの共有 scale/rotation があれば**それを流用**し、offset のみ合わせる。
  * 無ければ bbox ベースのフォールバック → 手動確認へ。

### 将来の構図対応

「特記事項」の以下のパターンは、回転（rotation）対応に加え、最終手段の Python ランドマーク（4.5）を要する可能性がある。段階導入とする。

* 体の向きが横（箒を持って顔だけこちらを向く 等）
* 違う体勢（椅子に座る、机に頬杖、布団に寝ている、横向きとのコンビネーション）

## 9. キャッシュとパフォーマンス

* マスコット側の顔情報・顔クロップは **outfit/pose ごとに1回**算出してキャッシュ。
* 共有変換 A も**シート単位で1回**確立してキャッシュ。
* OpenCV.js wasm（8〜10MB）は初回ロードでキャッシュ。
* 全工程ローカルのため API 課金・レイテンシ・非決定性の問題は発生しない。

---

## 10. テスト・自動評価

テスト資産（後述）の正解データを「眺めるだけ」でなく**目的関数・回帰指標**として活用する。

* 誤差指標：自動合成結果 vs `expr_*_OK.png` の**顔領域 IoU**、または重畳画素差。
* これにより、残るパラメータ（マスク楕円率、フェザー量など）を「測定された値」として調整できる。
* `__tests__/expression-alignment/assets/` の各 outfit をケースとして CI 化し、リグレッションを検出する。
* 既存テスト（`alignment-calculator.test.ts` ほか）は新アーキテクチャに合わせて再構成する。

## 11. プログラム構成

* 言語：TypeScript（renderer 側 / OpenCV.js wasm）。
* 画像処理：OpenCV.js を主とし、既存 Canvas API 資産（`content-bounds-detector` `background-masker` `feature-island-detector`）を流用。
* サーバ側 Python（anime-face-detector）は**オプション・最終手段**のみ（既定では導入しない）。
* 顔位置検出ライブラリは OpenCV.js（wasm）を採用。クラウド AI（Gemini 等）は**ローカルonly方針のため既定で不使用**。

### 11.1 マルチプラットフォーム実行戦略

今後 Web版・iOS/Android版 も予定している。本処理は**自前の OpenCV 計算であり AI 従量課金が発生しない**こと、**マスコット設定時の1回限り操作でホットパスではない**ことから、**完全 on-device（端末内実行）を全プラットフォーム共通の方針**とする。

* **実行場所：on-device（wasm）に統一。** サーバ実行は当面不要（インフラ費・要ネットワーク・画像アップロードを回避できる）。
* iOS/Android は **Web系スタック（Capacitor / React Native + WebView / Flutter + WebView / PWA）で構築する**前提が確定済み。WebView 内で wasm が動くため、**デスクトップ(Electron renderer)・Web版・モバイルの全プラットフォームで OpenCV.js コアを100%共有**できる。

| プラットフォーム | 実行場所 | コア共有 |
|---|---|---|
| デスクトップ(Electron) | renderer wasm | ◎ 同一TSコア |
| Web版 | ブラウザ wasm | ◎ そのまま |
| iOS/Android（Web系スタック） | WebView 内 wasm | ◎ そのまま |

### 11.2 コア／アダプタ分離（移植性の担保）

将来の純ネイティブ化や低スペック端末対応に備え、計算コアと実行環境を疎結合に保つ。コアは触らず I/O アダプタのみ差し替え可能とする。

```text
[共有] solveTransform(顔情報, スプライト) → { scale, rotation, offset, confidence }   ← 純粋・移植可能
          ▲
   ┌──────┴──────┐
[wasm adapter]   [server adapter（将来オプション）]
 (browser/webview) (Node + OpenCV)
```

* 既定は wasm アダプタのみ実装する。
* サーバアダプタ（Node + OpenCV）は、将来「純ネイティブ実装」や「低スペック端末のオフロード」が必要になった場合のフォールバックとしてのみ追加する（同一コアをホストするだけで二重実装にならない）。

### 11.3 独立ライブラリ化（packages 配下）

本処理は **UI(Vue)/Server(Express) から切り離した独立ライブラリ**として実装する。リポジトリは `ui/` と `server/` の両方から参照する必要があるため、**ワークスペースパッケージ `packages/expression-alignment` を新設**する。

* **framework 非依存**：Vue / PrimeVue / Electron IPC / Express / ファイル保存 / `MascotData` 永続化を一切持たない。
* **DOM 非依存**：`document.createElement('canvas')` 等への直接依存を排除し、画像デコード/ピクセル取得を `ImageInput` インターフェース（注入）として抽象化する。コアは `ImageData` 相当（幅・高さ・RGBA配列）だけを受け取る純粋関数とし、Canvas/wasm ロードはアダプタ側に置く。
* **決定論的**：AI 非依存・乱数なし。同入力→同出力。

**ライブラリが持つもの／持たないもの**

| 持つ | 持たない |
|---|---|
| `solveTransform`（変換の数学）、OpenCV.js レジストレーション、特徴島クロスチェック、マスク生成、confidence 算出 | Vue/PrimeVue、Electron IPC、Express ルート、ファイル保存、`MascotData` 永続化 |

**入出力契約（型で固定）**

```text
solveTransform(input: {
  baseImage:   ImageInput,            // マスコット画像（顔クロップ）
  sprite:      ImageInput,            // 通常 or 対象スプライト
  faceRegion?: BoundingBox,           // 既知なら渡す（無ければ内部で推定）
  sharedTransform?: SharedTransform,  // 単体モードでキャッシュ済みの共有変換A
}): AlignmentResult                   // { offsetX, offsetY, scale, rotation, confidence, mask }
```

UI/Server はこのパッケージを import して呼ぶだけ。既存 `ui/src/skills/expression-alignment/` の資産（`content-bounds-detector` `background-masker` `feature-island-detector` `alignment-calculator` `face-region-detector`）は本パッケージへ移設・改良する。

**パッケージ構成**

```text
packages/expression-alignment/        # framework / DOM 非依存ライブラリ
  src/
    solve-transform.ts                # 純粋コア（A/B分離・2モード）
    registration-opencv.ts            # ORB + estimateAffinePartial2D + ECC
    feature-island-detector.ts        # 改良版（目中心のみ・クロスチェック）
    mask.ts                           # 楕円フェザーマスク
    image-input.ts                    # ImageInput インターフェース（DOM/Node 非依存）
    types.ts
  adapters/
    canvas-browser.ts                 # ブラウザ/WebView 用 ImageInput 実装
    canvas-node.ts                    # テスト/Node 用（canvas npm）
  __tests__/
    solve-transform.test.ts           # ユニット（画像なし）
    golden-iou.test.ts                # 実画像 IoU 回帰
```

### 11.4 テスト戦略（2層）

「実行結果はテストで確認」を、目視ではなく**数値しきい値の自動判定**として成立させる。

1. **ユニット（高速・画像なし）**：対応点/ランドマークを直接与え、`solveTransform` の数学（相似変換の閉形式・クランプ・A/B分離・2モード分岐）を検証。
2. **ゴールデン/結合（実画像）**：`__tests__/expression-alignment/assets/` の実画像に **OpenCV.js wasm を Node で実行**（wasm は Node でも動作）。自動合成結果 vs `expr_*_OK.png` の **顔領域 IoU をしきい値判定**し、`expr_*_trimmed.png` はマスク評価に使用。

**合成画像の PNG 出力（目視確認・必須）**：数値判定（IoU）に加え、`npm run test` 実行時に**マスコット＋位置合わせ後スプライトを合成した画像を `__tests__/result/` に PNG 出力**する（例：`expr_◯◯_synthesized.png`）。`canvas-node` アダプタの `savePng(image, path)` を用いる。これにより合成結果を目視で確認できる（旧 `ui/.../visual-alignment.test.ts` が `result/` に出力していた仕組みの正式移植版。ただしハードコードパス・eval require は持ち込まない）。`__tests__/result/` は gitignore する。

Node 側の画像デコードは既に利用中の `canvas` npm を `canvas-node` アダプタで用いる。

## 12. 実装ステップ（段階導入）

ライブラリ（`packages/expression-alignment`）をテスト駆動で先に完成させ、UI/Server からは最後に呼び出す順で進める。

1. **パッケージ新設**：`packages/expression-alignment` をワークスペースに追加。`types.ts` と `image-input.ts`（`ImageInput` インターフェース）、`canvas-node` / `canvas-browser` アダプタの雛形を用意。
2. **既存資産の移設**：`ui/src/skills/expression-alignment/` の各モジュールをパッケージへ移設し、**DOM 直接依存（`document.createElement`）を `ImageInput` 経由に置換**。
3. **`solveTransform` 本体（実装済み）**：純粋関数として A/B 分離・confidence・2モード分岐を整備。画像→対応点（レジストレーション）は `RegistrationProvider` として注入分離し、対応点→相似変換の数学（`similarity.ts`）をユニットテスト済み。戻り値はピクセル空間の `SimilarityTransform`（エディタ座標変換は step9）。
4. **OpenCV.js レジストレーション（実装済み）**：ORB + BFMatcher + ratio test + `estimateAffine2D`（RANSAC インライア検出）を `registration-opencv.ts` に実装し、`RegistrationProvider`（`{ pairs, inlierRatio }` を返す）として `solveTransform` に注入。**ロードは環境別アダプタ**：Node/テストは `opencv-wasm`（同期・ハング回避）、ブラウザ/WebView 本番は `@techstark/opencv-js`（`onRuntimeInitialized`）。`@techstark` を Node で await するとハングする既知問題があるため、テストでは使わない。
5. **特徴島クロスチェック**：feature-island-detector を**目中心のみ**に縮退させクロスチェック専用化（コメントと実値の不整合も解消）。
6. **楕円フェザーマスク**：`mask.ts` を実装し共有変換 A に統合。
7. **ゴールデン IoU テスト ＋ 合成画像出力**：`expr_*_OK.png` との顔領域 IoU をしきい値判定する回帰テストを整備し、CI 化。**併せて `npm run test` で合成画像（マスコット＋位置合わせ後スプライト）を `__tests__/result/` に PNG 出力**し目視確認可能にする（`savePng` を使用。出力先は gitignore）。
8. **`MascotAsset` 拡張**：`rotation` を追加（後方互換）。レンダラ/エディタ（`ExpressionEditorModal.vue`）に反映。
9. **UI/Server 結線**：UI から本パッケージを呼び、2実行モード（16枚 / 単体）と confidence ゲート → 手動エディタへのプリセット連携を実装。
10. **（必要時のみ）** Python anime-face-detector を横向き等の最終手段として追加検討。サーバアダプタも必要になった時点で追加。

## 13. 特記事項

マスコット画像は一般に正面立ち/バストアップだが、下記パターンも将来的に考慮が必要。

* 体の向きが横：体が横を向き、顔がすこしこちらを向いている（例：箒を持って顔だけ少しこちらを向く）
* 違う体勢：座っている、布団に寝ている（椅子に座る、机に頬杖をついてこちらを見る、横向きとのコンビネーション）

これらは rotation 対応（第6章）と、必要に応じ Python ランドマーク（4.5）で段階的に対応する。
