# 音声合成（TTS）向け台詞抽出の自動最適化設計書 (to-tts-transform-plan.md)

本ドキュメントでは、LLM の応答が「ロールプレイモード（ト書き＋セリフの複合文章）」か「一般相談モード（平文、または単なる強調用括弧）」かをシステム側で自動判定し、ユーザーが手動で設定を切り替えることなく、TTS（音声合成）へ最適なテキストを引き渡すための設計および実装計画を定義します。

---

## 1. 現状の課題と原因

現在、キャラクター設定で「地の文を読み上げない（`ttsReadNarrative === false`）」に設定している場合、`filterDialogue` 関数によってト書きや地の文をカットする処理が行われます。しかし、AIの応答形式が対話のコンテキストによって異なるため、以下の不都合が発生しています。

### ロールプレイ（なりきり）モード時の挙動
* **特徴**: AIは `（微笑みながら）「こんにちは！ 調子はどう？」` や、`「準備できたよ！」と嬉しそうに駆け寄る。` のような、ト書き（行動描写）とセリフが混ざった小説風 of テキストを出力します。
* **期待されるTTS処理**: 丸括弧内の行動描写や、括弧外の `と嬉しそうに駆け寄る` といったナレーション文（地の文）をカットし、セリフ部分（`こんにちは！ 調子はどう？` や `準備できたよ！`）のみを読み上げること。
* **現状**: `dialogue-filter.ts` の引用判定ルールが機能し、期待通り動作します。

### 一般相談・解説モード時の挙動
* **特徴**: AIは `はい、了解です！「タスクの登録」を実行しますね。` のように、ナレーションではなく、平文のセリフの途中に単語の強調目的で `「」` などの括弧を使用します。
* **期待されるTTS処理**: 括弧の外側にある `はい、了解です！` や `を実行しますね。` もすべてマスコットのセリフであるため、括弧記号のみを外して**全体を読み上げる**こと。
* **現状**: 括弧の外側に文字（`はい、了解です！` など）が存在するため、プログラムが「ト書き（地の文）が存在する」と誤判定し、外側のセリフをすべて切り落として `タスクの登録` だけを読み上げてしまう（その他の文章が飛ばされる）現象が発生します。

---

## 2. 解決アプローチの検討

この問題を解決するため、以下の 2 つのアプローチ、およびそれらを組み合わせたハイブリッド方式を検討します。

### アプローチ A: プログラム側での「ナラティブ度（ト書き度）」自動スコアリング判定
AIからの返答テキストの特徴を分析し、「ロールプレイ（小説形式）」か「一般対話」かを動的にスコアリングして判定します。

* **判定ロジック案**:
  以下のいずれかの条件を満たす場合、テキストを **「ロールプレイモード」** と判定し、厳密なト書きカット（括弧外のテキストの切り落とし）を行います。それ以外は **「一般対話モード」** とみなし、括弧記号のみを除去して全体を読み上げます。
  1. **ト書き記号の存在**: 丸括弧 `（）` `()` や アスタリスク `*` で囲まれたテキスト（行動描写）が 1 つ以上含まれている。
  2. **引用パターンの検出**: `」と`、`」って`、`」と言いながら` のように、閉じ括弧の直後に三人称・ト書き特有の引用表現が存在する。
* **メリット**: LLM側の出力に多少の揺れがあっても、プログラム側で安全にフォールバックできます。
* **デメリット**: 完全に記号がないロールプレイ文（例: `「待って！」マスコットは引き止めた。` などの `」と` を伴わない形式）で、切り落としが機能しない可能性があります。

### アプローチ B: システムプロンプトによる出力フォーマットの厳密制御
LLM へのシステムプロンプト（ツールガイドライン）に、出力形式の厳格なルールを追加します。

* **プロンプト指示案**:
  > あなたは会話（セリフ）以外の行動描写やト書きを出力する際は、必ず丸括弧 `（行動描写）` または `*行動描写*` で囲んで出力しなければなりません。括弧で囲まれていないテキストは、すべてあなたが口頭で発話するセリフとして解釈されます。
* **メリット**: AIが正しく従う限り、プログラム側での誤判定が一切なくなり、最も確実なテキスト分割が可能になります。
* **デメリット**: 軽量なローカルLLMなど、プロンプトの指示遵守率が低いモデルの場合、指示を無視して括弧外に平文のト書きを出力してしまい、それが読み上げられてしまうリスクがあります。

---

## 3. 推奨設計案：ハイブリッド方式

プログラム側での**「ナラティブ度自動スコアリング（アプローチ A）」**をベースにしつつ、**「プロンプトによる出力ルールの補強（アプローチ B）」**を補助的に組み合わせるハイブリッド方式を推奨します。

### 1. プログラム側 (`filterDialogue`) の改修設計
`filterDialogue` 関数内に、ナラティブ特性を評価するロジックを導入します。

```typescript
export function filterDialogue(text: string): string {
    if (!text) return '';

    // A. ロールプレイ（ナラティブ）判定用のフラグをテキスト全体から検出
    // 1) 行動描写の括弧（（）、()、* *）が本文中に存在するか
    const hasActionNarrative = /（[^）]*）|\([^)]*\)|\*[^*]+\*/.test(text);

    // 2) 台詞引用パターン（」と、」と言った、等）が存在するか
    const hasQuoteFollowUp = /[」』]\s*(と|って|と言|と呟|と囁|と叫|と微笑)/.test(text);

    // ロールプレイ・小説ト書き形式であると判定する境界条件
    const isRolePlayMode = hasActionNarrative || hasQuoteFollowUp;

    // B. 行動描写の除去（丸括弧、アスタリスク部分をカット）
    let cleanText = text;
    cleanText = cleanText.replace(/（[^）]*）/g, '');
    cleanText = cleanText.replace(/\([^)]*\)/g, '');
    cleanText = cleanText.replace(/\*[^*]+\*/g, '');

    const hasJapaneseBrackets = /[「『]/.test(cleanText);

    if (hasJapaneseBrackets) {
        if (isRolePlayMode) {
            // 【ロールプレイモード】
            // 括弧の外側にト書き（地の文）が存在するため、括弧で囲まれたセリフ部分のみを抽出して結合する
            const regex = /[「『](.*?)[」』]/g;
            let match;
            const parts: string[] = [];
            while ((match = regex.exec(cleanText)) !== null) {
                if (match[1].trim()) {
                    parts.push(match[1].trim());
                }
            }
            if (parts.length > 0) {
                return parts.join(' ');
            }
        } else {
            // 【一般対話モード】
            // 括弧は単なる単語の強調や表現であるため、括弧記号のみを除去して外側のテキストも含めてすべて残す
            return cleanText.replace(/[「」『』]/g, '').trim();
        }
    }

    // 括弧がない場合は、丸括弧が除去された残りのクリーンテキスト全体を読み上げる
    cleanText = cleanText.replace(/\n\s*\n+/g, '\n');
    return cleanText.trim();
}
```

### 2. システムプロンプト（ガイドライン）の更新
`chat-ai-service.ts` 内の `toolUseGuideline` またはシステムプロンプトの基本ルールに以下の一文を追加します。
> 「セリフ以外の行動描写や状態説明を出力する場合は、必ず丸括弧 `（）` または `* *` で囲んでください。それ以外のテキストはすべて口頭で話すセリフとして解釈されます。」

## 4. 検証・テスト計画

本設計の導入にあたり、以下のテストケースで `filterDialogue` の挙動を検証します。

| 入力テキスト | 期待されるTTS出力テキスト | 判定モード | 目的 |
| :--- | :--- | :--- | :--- |
| `はーい、了解です〜！「〇〇を登録したよ！」` | `はーい、了解です〜！〇〇を登録したよ！` | 一般対話 | 括弧外のセリフ切り落とし防止（波ダッシュ維持） |
| `（微笑みながら）「おはよう」と手を振る。` | `おはよう` | ロールプレイ | 括弧外のト書き除去 |
| `「待って！」*走って追いかける*` | `待って！` | ロールプレイ | アスタリスクによるト書きの除去とセリフ抽出 |
| `「時間になったもの」を自動で消去します。` | `時間になったものを自動で消去します。` | 一般対話 | 単語の強調用括弧の除去と全体維持 |
