# 修正内容の確認手順（Walkthrough）

設定ファイル保護機構および SQLite データベース導入（テーブル・API完全分離版）の動作確認手順です。

---

## 1. サーバー側: SQLite テーブル分割と自動移行の確認

### 1.1. 自動マイグレーション（インポート）の確認
1. 修正前の既存ユーザー設定ファイル `storage/users/<userId>/config.json`（`mascots` 配列を含む巨大な設定）が存在することを確認します。
2. サーバー（またはアプリケーション）を起動します。
3. `app/storage/mascot_app.db` ファイルが新規生成されていることを確認します。
4. SQLite 閲覧ツールを用いて、以下のテーブルを検証します：
    * `user_configs` テーブル: 該当ユーザーのシステム設定データ（音量やAPIキーなど）が保存され、**`mascots` 配列が含まれていないこと**を確認します。
    * `user_mascots` テーブル: 各マスコット（服装や表情、ポーズのアセット情報など）が個別のレコードに分解して保存されていることを確認します。
5. 既存の設定ファイルが `storage/users/<userId>/config.json.imported` にリネームされ、二重インポートが防がれていることを確認します。

### 1.2. API の完全分離（フェーズB）確認
1. ブラウザのデベロッパーツール（ネットワークタブ）または API テストツールを用いて、以下のエンドポイントを検証します：
    * **GET `/api/config`**: レスポンスされる JSON 内に `mascots` プロパティが存在せず、軽量なシステム設定のみが返却されることを確認します。
    * **GET `/api/mascots`**: ユーザーに紐づくマスコットの一覧（アセット情報などを含む）が配列として返却されることを確認します。
    * **POST `/api/mascots/:id`**: 特定マスコットのデータを送信し、DBの `user_mascots` テーブルの該当レコードのみが更新（`updated_at` が更新）されることを確認します。

## 2. クライアント側（Electron / フロントエンド）の動作確認

### 2.1. 通常保存とバックアップ生成の確認
1. アプリケーションを起動し、一度でも設定の更新（または正常ロード）が行われた後、設定保存先ディレクトリに `config.json.bak` が生成されていることを確認します。
2. 保存された `config.json` および `config.json.bak` のファイルサイズが極めて軽量（数KB以下）であり、**`mascots` 配列のデータが含まれていないこと**を確認します。

### 2.2. メインファイル破損時の自動復元確認
1. アプリケーションを終了します。
2. 本番の `config.json` の中身をテキストエディタ等で完全に空にするか、不正な JSON 文字列（例: `{ "invalid`）に変更して壊します。
3. アプリケーションを起動します。
4. コンソールに `[Config] Failed to load config file` というパース失敗のエラーログが出力され、続いて `[Config] Successfully restored config from backup.` というバックアップからの復元ログが出力されることを確認します。
5. アプリケーションが初期化されることなく、元の設定を維持して正常に起動することを確認します。

### 2.3. 完全破損時の保護（読み込み専用モード）確認
1. アプリケーションを終了します。
2. 本番の `config.json` とバックアップの `config.json.bak` の両方を完全に壊します（または削除します）。
3. アプリケーションを起動します。
4. アプリケーションは初期設定（`defaultData`）で起動します。\r
5. 起動後、設定を変更しようとしても、自動保存や手動保存によってファイルが初期値で上書きされないこと（読み込み専用モードとして動作し、元の破損状態を勝手に初期値で上書きして消し去らないこと）を確認します。\r

## 3. サーバー側: 会話履歴の SQLite 化 (`chat_histories.db`) と自動移行の確認\r
\r
### 3.1. 会話履歴の自動マイグレーションの確認\r
1. アプリケーション起動（または API 呼び出し）前に、該当ユーザーのフォルダ `storage/users/<userId>/chat_history.json` が存在することを確認します。\r
2. サーバーを起動し、対象ユーザーでアプリにアクセスします（これにより `/api/history` の GET リクエストが送信されます）。\r
3. `storage/users/<userId>/chat_histories.db` が新規生成されていることを確認します。\r
4. SQLite 閲覧ツール等で以下のテーブルを確認します：\r
    * `chat_sessions` テーブル: 過去のセッションIDとタイトル、タイムスタンプが保存されていること。\r
    * `session_participants` テーブル: 各セッションに対し、ユーザーID（`userId`）とマスコットID（`mascotId`）の参加レコードが存在すること。\r
    * `messages` テーブル: すべての対話メッセージが、正しい送信元（`user` または `mascot`）および日時情報とともに保存されていること。\r
5. 移行元の JSON ファイルが `storage/users/<userId>/chat_history.json.imported` にリネームされていることを確認します。\r
\r
### 3.2. メッセージの保存とセッションクリーンアップの確認\r
1. チャットパネルからマスコットに新しいメッセージを送信します。\r
2. DBの `messages` テーブルに該当メッセージが即座に追記されることを確認します。\r
3. フロントエンドでセッションを削除し、別のセッションを作成・切り替えます。\r
4. DBの `chat_sessions` および外部キー制約 (ON DELETE CASCADE) を通じて `session_participants`, `messages` から、削除されたセッションに紐づく古いレコードが完全に抹消されていることを確認します。\r
