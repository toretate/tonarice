対象仕様: `docs/specs/マスコット画像取扱仕様書.md` 第6章（課題）・第7章（改訂設計）

## 0. ゴールと非ゴール

**ゴール**
- 画像インポート経路を「即時アップロード → パスのみ保持」に統一し、Base64 を配線・永続の主役から降ろす（推奨A）。
- キャッシュバスターをグローバル版数からアセット単位へ移行する（推奨B）。

**非ゴール（今回スコープ外）**
- 内容アドレスストレージ / 重複排除（推奨C）。別Issueとする。
- ストレージのDB化、CDN配信。
- 既存 `mascot_config.json` のスキーマ破壊的変更。後方互換を維持する。

**互換性の原則**
- サーバー側の `processMascotAssets` によるBase64デコードは **削除しない**（フォールバックとして残す）。これにより既存の保存済みデータ・古いクライアントからのPOSTが壊れない。

## フェーズ1: インポート即時アップロードへの統一（推奨A）

### 1-1. 共通アップロードヘルパの新設
- 追加: `useMascotSettings.ts`（または `mascots/` 配下の新ユーティリティ）に
  `uploadImportedImage(mascotId, assetType, assetId, source): Promise<string /* /mascots/... path */>`
- 中身: `window.electronAPI.saveMascotImage` を呼ぶ既存経路に集約。戻り値のパスを返す。失敗時は例外を投げ、呼び出し側でBase64フォールバック（従来挙動）に退避。
- ファイル名規約は既存に合わせる: `${assetType}/${assetId}.${ext}`（拡張子はMIMEから決定。`save-image.post.ts` は現状クライアント指定 `filename` に従う）。

### 1-2. 未対応インポート経路の差し替え
以下を「取得 → 即アップロード → パス格納」に変更する。

- [ ] `MascotSettings.vue addOutfitImage` (:112)
  - `result.path`（Base64）を直接格納せず、`uploadImportedImage(..., 'outfits', newOutfit.id, result.path)` の戻りパスを `newOutfit.path` に設定。
  - `editingMascot.id` が未確定（新規マスコット未保存）の場合の順序に注意 → 「1-4」参照。
- [ ] `MascotSettings.vue handleCropNew` (:242) のクロップ元画像
  - `cropImageSrc` はプレビュー用途なので ObjectURL 化（フェーズ3）と合わせて検討。最低限、確定時の元画像保存は `handleCropDone` 既存ロジックを流用。
- [ ] アバターインポート経路（`MascotSettings.vue:244` 付近を確認して該当すれば差し替え）

### 1-3. プレビュー用の一時表示
- 即アップロードが成功すれば `/mascots/...` パスで描画できるため、原則プレビューもサーバー画像を使う。
- ネットワーク往復のちらつきが問題になる場合のみ、アップロード完了までの短時間 ObjectURL を表示（フェーズ3で `URL.createObjectURL` 化とセットで対応）。

### 1-4. 新規マスコット（id未確定）時の順序保証
- `save-image` は `mascotId` を必須とする。新規マスコットで `editingMascot.id` が空の場合、
  - 対応: マスコット生成時点で id を採番済みにする（既存 `addMascot` を確認）。既に採番されていれば問題なし。
  - 未採番なら、初回だけ従来のBase64フォールバックで全体POSTに委ね、保存後に id 確定 → 以降は即アップロード、とする。

### 1-5. saveConfig ペイロードの軽量化確認
- フェーズ1完了後、`config.ts:697` のPOSTボディに `data:image/` が原則含まれないことをログ/テストで確認。
- ボディは引き続き `mascots` 全体を送るが、中身はパスのみになる（構造は非破壊）。将来的な差分POST化は別Issue。

### フェーズ1の受け入れ条件
- 衣装・アバター・表情のインポート後、`editingMascot` 内の該当 `path` が即座に `/mascots/...` になる（Base64が残らない）。
- インポート直後に `save-settings` が発火しても、POSTボディにBase64が含まれない。
- Web版・Electron版の双方で新規インポート画像が描画される。
- 既存の保存済みマスコット（Base64混在の旧データ）を読み込んでも壊れない（フォールバック健在）。

## フェーズ2: キャッシュのアセット単位化（旧案・後段の再設計で置換）

### 2-1. アセット版数の導入
- 案1（推奨・低リスク）: アセットに `version`（保存時に採番する整数 or 保存時刻）を持たせ、`resolveImageUrl` の `?v=` をグローバル `configVersion` から **そのアセットの version** に変更。
- 案2（理想・やや大工事）: 保存ファイル名に内容ハッシュを含め immutable 化。`[...path].ts` のレスポンスに `Cache-Control: public, max-age=31536000, immutable` を付与。パス変更に伴い旧ファイルのGCが必要。
- まず案1で着手し、効果測定後に案2を検討。

### 2-2. 変更箇所
- [ ] `useMascotSettings.ts:91 resolveImageUrl` … `?v=` の値をアセット版数へ
- [ ] `MascotViewer.vue` の `resolveImageUrl` … 同上（2箇所の実装を共通化できるか検討）
- [ ] 保存時に変更アセットの version のみ更新（`saveMascotImage` 成功時 or サーバー応答で採番）
- [ ] `config.ts:313 configVersion++` … 画像キャッシュ目的での全体インクリメントを廃止（他用途があれば残す。要調査）

### 2-3. サーバー側キャッシュヘッダ
- `[...path].ts` の `Cache-Control` を版数戦略に合わせて調整（案1なら短めmax-age維持＋クエリ版数、案2ならimmutable）。

### フェーズ2の受け入れ条件
- 1アセットのみ変更・保存した際、他アセットの画像リクエストが 304/キャッシュヒットになる（DevTools Network で確認）。
- 変更したアセットは新版数で再取得される。

## フェーズ2 再設計（2026-07-13 クロスモデルレビュー反映）

フェーズ2の当初案（推奨A: アセット単位 `?v=` / 推奨B: 内容ハッシュ）について、Antigravity(Gemini Pro)にセカンドオピニオンを依頼し、その指摘をClaude側で実コード検証した結果、**当初案の前提に誤りと過小評価があった**ため方針を見直す。

### 検証で判明した事実（当初案の前提の訂正）

1. **`resolveImageUrl` 相当のキャッシュバスター（`?v=${configVersion}`）は「2箇所」ではなく13ファイル以上に散在**している。
   - `MascotViewer.vue` / `useMascotSettings.ts` / `WindowSettingsPanel.vue` / `IntegratedLayout.vue` / `MascotVerticalList.vue` / `AiExpressionGeneratorModal.vue` / `EmotionAssignmentModal.vue` / `ExpressionAnimationEditor.vue` / `ExpressionEditorModal.vue` / `MascotOutfitSettngs.vue` / `ImageCropModal.vue` / `expression-editor/Step1–3`。
   - → 当初「推奨A」はこの13ファイル全てをアセット単位版数参照に書き換える必要があり、想定よりコスト・リスクが大きい。
2. **`configVersion` の用途はキャッシュバスターのみ**（grep で全用途を確認）。他のリアクティブ再描画等に使われていないため、キャッシュ戦略の変更で他機能を壊す懸念はない。
3. **配信ルートは既に `Cache-Control: public, max-age=3600` を設定済み**（`app/src/server/routes/mascots/[...path].ts:105`）。「ヘッダー未設定」ではない。
   - この `max-age=3600` のため、**編集直後1時間はブラウザが条件付きリクエストすら送らず古い画像を出す**現象の一因になっている。ETag/条件付きGETを効かせるには、この値を `no-cache`（= `max-age=0, must-revalidate`）へ変更し毎回検証させる必要がある。
4. **PixiJS `Assets.load` は URL 文字列でキャッシュする**。`?v=` を変えれば再取得はされるが、古い URL のテクスチャは `Assets.unload(旧URL)` を明示的に呼ばないとメモリに残る（リーク）。どのキャッシュ戦略を採っても別途対応が必要（`MascotViewer.vue` の現状 load/unload 実装を要確認）。

### レビュー時の第一候補 = 案C（ETag / Last-Modified、サーバー主導）

**方針**: クライアント側の版数管理をやめ、サーバー配信ルートでファイルの `mtime` に基づく `Last-Modified` / `ETag` を返し、ブラウザの条件付きリクエスト（`If-None-Match` / `If-Modified-Since`）で 304 を返す。

- **変更が `[...path].ts` 1ファイルにほぼ閉じる**。13ファイルのクライアント改修・`mascot_config.json` スキーマ変更・保存応答の版数配線がすべて不要。
- **必須の同時変更**: `Cache-Control` を `public, max-age=3600` → `no-cache`（または `max-age=0, must-revalidate`）に変更。これをしないと ETag が発火せず現行の「1時間ズレ」も残る。
- **トレードオフ**: アセット毎に軽量な条件付きGET（304）が1往復発生する。Electron のループバックでは無視できる。リモートWebでも小さいが、純キャッシュヒット（リクエスト0）にはならない。
- **付随整理（任意）**: 版数が不要になるので、各ファイルの `?v=${configVersion}` 付与は段階的に撤去可能（撤去は必須ではない。残っていても 304 は機能する）。

### 代替: 案A'（アセット単位 `?v=` ＋ resolveImageUrl 単一化）

純キャッシュヒット（リクエストゼロ）の性能が要件になる場合の代替。

- 前提として **`resolveImageUrl` を単一の composable/util に集約**し、13ファイルの重複を解消する（これ自体が独立した価値のあるリファクタ）。
- アセットに `version`（保存時刻等）を持たせ、**サーバーが `save-image` 成功応答で採番して返す**（クライアント楽観更新はレース＝404/旧画像取得の原因になるため避ける）。版数は `mascot_config.json` に保存して再起動後も永続。
- URL 版数でファイルは実質 immutable 化できるため、配信ヘッダーは `Cache-Control: public, max-age=31536000, immutable` に強化可能。
- コスト大（13ファイル改修＋スキーマ＋応答配線）。ETag 案で要件を満たせるなら不要。

### 案B（内容ハッシュ＋immutable）は当面見送り

古いファイルのGCが必要でスコープ過大。将来ストレージ最適化を行う場合の選択肢として保留。

### フェーズ2の作業順（改訂）

1. `MascotViewer.vue` の PixiJS 現状 load/unload を確認（`Assets.unload` 要否の確定）。
2. 案C（ETag/Last-Modified）を `[...path].ts` に実装 ＋ `Cache-Control` を `no-cache` 系へ変更。
3. 検証: 1アセットのみ変更・保存時、他アセットが 304（キャッシュ有効）、変更アセットのみ 200 再取得になること（DevTools Network）。編集直後の表示ズレが解消すること。
4. 効果測定。純ヒット性能が必要なら案A'（`resolveImageUrl` 単一化＋アセット版数）へ拡張。

### フェーズ2の受け入れ条件（改訂）

- 1アセット変更・保存時、他アセットの再取得が発生しない（304 かキャッシュヒット）。
- 変更したアセットは即座に新しい内容で表示される（編集直後の1時間ズレが解消）。
- Web版のリモートアクセス時も認可（403/200）が維持される。
- PixiJS のテクスチャがアセット差し替え時にリークしない。

## フェーズ2 実装結果（2026-07-14）

Geminiサブレビューで、ETagだけではPixiJSのURLキャッシュを更新できないこと、およびクエリを除去して比較する既存watchでは画像版数の変更を検出できないことが判明した。このため、最終的に **案A'のアセット固有URLと案Cの条件付きGETを併用**した。

### 採用した設計

- 保存API/IPCが成功時に `?v=<UUID>` を付けたパスを返し、そのパスを既存のアセット `path` として永続化する。スキーマ項目の追加は行わない。
- `resolveMascotImageUrl` を共通ユーティリティ化し、散在していたグローバル `configVersion` の付与を除去する。
- `MascotViewer.vue` はクエリを含む完全URLを監視し、ロード世代が古い結果を描画へ適用しない。
- 表示中・先読み中のURL集合を追跡し、参照されなくなったURLのみ `Assets.unload` する。
- 配信ルートは認証・所有者確認後に `ETag` / `Last-Modified` を評価し、`Cache-Control: no-cache` と304応答を提供する。`If-None-Match` は `If-Modified-Since` より優先する。

### 完了タスク

- [x] Web即時保存、Base64互換保存、Electron IPCの全保存経路でアセット固有版数を採番
- [x] URL解決処理の共通化とグローバルキャッシュバスターの廃止
- [x] PixiJSの完全URL監視、ロード競合防止、旧テクスチャ解放
- [x] ETag / Last-Modified / no-cache / 304と認証順序の実装
- [x] URL解決、保存応答、HTTP条件付きGET、版数変更時再ロードの単体テスト追加

### 検証基準

- 変更アセットの保存応答だけが新しい版数URLになり、他アセットのURLは不変である。
- 同一URLへの条件付きGETは304、版数が変わったURLはPixiJSで再ロードされる。
- `If-None-Match` がある場合は `If-Modified-Since` より優先され、認証失敗時はETag一致でも401/403になる。
- 旧ロードが新ロードより遅く完了しても、旧テクスチャが表示へ戻らない。
- 表示中または先読み中のURLは解放せず、両方から外れたURLとアンマウント時の所有URLを解放する。
