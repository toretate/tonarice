# Geminiによる実装レビュー

Geminiへ読み取り専用の独立レビューを依頼し、Codexが指摘を再確認して最終検証する。

## 1. 検証・レビューの手順

Codexから作業完了の報告を受けた、または差分が発生した場合は、以下の手順で安全に検証を行います。

### ステップ 1: `git diff` による差分の安全な確認
- **変更点の把握**: 修正されたファイルと差分（diff）を確認します。
- **コード内の簡易静的スキャン（セキュリティチェック）**:
    - 変更されたコードに、不審な外部通信（例: `fetch('http://malicious.com')`）、ファイルの不正な削除・上書き（例: `fs.rmSync`）、不適切なシェルコマンドの実行（例: `exec('rm -rf ...')`）が混入していないか目視および簡易チェックを行います。

### ステップ 2: Geminiへ読み取り専用レビューを依頼
- Antigravityのラッパーを `-Tier pro -Mode plan -Sandbox -Digest -Dir <repo>` で実行する。
- 重要度付き findings、根拠となる `file:line`、影響、修正案、テスト不足、妥当な点を要求する。
- ファイル編集、コミット、外部送信を禁止し、機密情報をプロンプトへ含めない。

### ステップ 3: Codexによる再確認と検証
- Geminiの指摘を差分とソースで再確認し、誤検知を除外する。
- 開発サーバーやElectronを停止せず、別プロセスで `app/` の `npm run lint`、`npm run typecheck:all`、変更箇所に近い対象テストを実行する。
- Geminiの自己申告を検証結果として扱わず、Codexが実際のコマンド結果を確認する。
- 自動コミット、push、merge、deployを行わない。

---

## 2. 検証レポートの標準フォーマット

検証が完了したら、結果を以下のフォーマットで整理し、必要に応じてCodexへの修正指示（フィードバック）を作成します。

```markdown
# 検証結果レポート (for Codex Review)

## 1. 検証概要
- **対象タスク**: [タスク名]
- **検証日時**: [日時]
- **検証ステータス**: [✅ 成功 / ❌ 失敗 / ⚠️ 警告あり]

## 2. セキュリティ・静的解析チェック
- [ ] 悪意あるコードや不要な外部通信コードの混入なし：[確認結果]
- [ ] 機密情報のハードコードなし：[確認結果]

## 3. テスト・コンパイル結果
- **TypeScript コンパイル**: [成功 / エラー内容]
- **テスト実行結果**: [例: 24 passed, 0 failed]
- **実行時間 / リソース消費**: [例: 2.3秒 / OOMの兆候なし]

## 4. フィードバック / 次のアクション
[テスト失敗時や改善点がある場合の、Codexに対する具体的な修正指示をここに記述]
```

---

## 3. セキュリティチェックリスト（テスト実行前確認）

テストコマンドを実行する直前に、必ず以下のチェックリストをエージェント自身で評価してください。

- [ ] 変更されたコード内に、不正なネットワークアクセスコードが追加されていないか？
- [ ] 実行するテストコマンドに引数インジェクションの脆弱性がないか？
- [ ] テスト実行時のタイムアウト設定（`--timeout` やシェル側のタイムアウト）が有効になっているか？
- [ ] テスト用ダミーデータに実際の認証情報（本番環境のトークンなど）が紛れ込んでいないか？
